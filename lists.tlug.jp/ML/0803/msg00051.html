Hi all,

trac[1] is a system for tracking  issue. It manage issues with
"ticket". Currently, we are using trac as a bug tracking system and
change requirment tracking. For the latter, extended ticket
workflow[2] is used.

The original word ( of manage requirements ) is Japanese is 課題管理. A
kadai may or maybe be an issue. It can be a feature we are involving,
the implementation of it is undefined, we may do it or may not do it.
When it is a feature, I think a ticket is too small in the context for
a kadai to put in. A kadai may involve the whole project and before we
implement it, we do not know what modules ( components ) of the
current code have a relationship with it.

Strictly speaking, trac is not a software configuration management
system because it lacks the feature I've mentioned.

I want to make trac be able to manage the 課題管理. Can I do it if I
customize trac's workflow. And how? ( See [2] for detail ).

[1] http://trac.edgewall.org/
[2] http://trac.edgewall.org/wiki/TracWorkflow
